### **유스케이스 작성 핵심 지침**

### 시스템이 액터에게 제공하는 '가치(Value)'와 '목적'에 집중하라
유스케이스는 단순한 기능의 나열이 아니라, 시스템이 액터의 목표를 달성하도록 돕고 액터에게 의미 있는 결과를 제공하는 과정입니다. 각 유스케이스는 단독으로 실행되었을 때 액터에게 **독립적이고 식별 가능한 가치**를 전달해야 합니다. "액터가 이 시스템을 통해 궁극적으로 무엇을 하려는 것인가?"를 항상 질문해야 합니다.

### 시스템의 '바운더리(Boundary)'와 책임의 한계를 명확히 설정하라
유스케이스 작성을 위해서는 시스템이 어디까지를 책임지고, 어디서부터가 시스템 외부(액터)인지를 먼저 정의해야 합니다. 시스템을 외부의 자극에 반응하는 블랙박스로 취급하여, 내부 로직보다는 **시스템과 외부 환경 간의 경계에서 발생하는 상호작용**을 찾아내는 것이 필수적입니다. 또한 장애나 예외 상황 발생 시 **"누가 고쳐야 하는가?"(내부 vs 외부)**를 기준으로 책임의 주체를 식별하십시오.

### 기능을 분해(Functional Decomposition)하지 마라
유스케이스는 시스템의 요구사항을 작고 파편화된 기능적 단위로 쪼개는 분석(Analytic) 기법이 아닙니다. 큰 문제를 작은 조각으로 나누면(예: '주문 생성', '주문 수정', '주문 삭제' 등으로 쪼개기) 전체 시스템이 무슨 가치를 제공하는지 알아볼 수 없게 됩니다. **단순한 데이터 조작(CRUD)에 불과한 것들을 억지로 유스케이스로 만들지 말고**, 여러 기능을 종합(Synthetic)하여 하나의 완결된 가치를 제공하는 큰 그림(Story)을 그려야 합니다.

### 대화식(Dialog) 구조로 상호작용을 기술하라
흐름은 액터와 시스템 간의 대화나 연극의 대본처럼 전개되어야 합니다. "액터가 어떤 행동(이벤트)을 하면, 시스템은 어떻게 반응한다"는 식의 **핑퐁(Ping-Pong)식 상호작용**을 명확하게 묘사해야 합니다. 행동의 초점이 액터에서 시스템으로(또는 그 반대로) 바뀔 때마다 단락이나 단계를 분리하는 것이 좋습니다.

### 완전한 문장을 사용하고 '주어'를 절대 생략하지 마라
문장은 행위의 주체가 누구인지 한눈에 알 수 있도록 **"시스템은(The system)..." 또는 "액터 [이름]은(The actor [Name])..."**으로 반드시 시작해야 합니다. 주어를 생략하면 해당 동작을 액터가 하는 것인지 시스템이 하는 것인지 모호해집니다.

### 능동태와 현재 시제를 사용하여 직설적으로 작성하라
수동태나 모호한 표현을 피하고, 신문 기사처럼 간결하고 직접적인 문체를 사용합니다. "~해야 한다(shall, should)"와 같은 미래/당위적 표현보다는 **현재 시제의 능동태**를 사용하여 "~한다(does)"로 명확히 서술해야 합니다. (예: "입력된 금액은 시스템에 의해 검증되어야 한다" (X) -> "시스템은 입력된 금액을 검증한다" (O))

### 내부 설계(How)나 구체적인 UI 묘사를 배제하라 (What에 집중)
유스케이스는 시스템이 '무엇(What)'을 하는지 설명하는 것이지, '어떻게(How)' 하는지 설계하는 곳이 아닙니다. "시스템이 드롭다운 목록을 팝업창으로 띄운다"와 같은 UI 세부 사항이나 내부 컴포넌트의 작동 방식은 제외하고, 시스템의 본질적인 행동에 집중하십시오. UI와 관련된 내용은 스토리보드나 프로토타입에 맡기는 것이 좋습니다.

### 기본 흐름(Basic Flow)을 우선 작성하고 예외 처리는 대안 흐름(Alternative Flow)으로 빼라
모든 것이 완벽하게 진행되는 가장 일반적인 '해피 패스(기본 흐름)'를 5~10단계 정도로 먼저 완성하세요. 실패 조건이나 선택적인 분기 조건들까지 기본 흐름에 한꺼번에 섞어 쓰면 가독성이 크게 떨어집니다. 이러한 변수들은 기본 흐름 밖으로 빼내어 별도의 **대안 흐름(Alternative Flows)** 섹션에서 다뤄야 핵심 로직을 잃지 않습니다.
80% 이상 일반적인 성공케이스를 기본 흐름으로 작성하고, 20% 이하의 예외케이스를 대안 흐름으로 작성하는 것이 좋습니다.


### 시작 조건 (Starting Conditions) 시스템이 아닌 '액터의 행동'으로 기술
유스케이스는 자발적으로 시작되지 않으므로, 흐름의 첫 단계는 반드시 액터가 발생시키는 이벤트로 명시합니다. *권장 형식:* "이 유스케이스는 액터가 [어떤 행동]을 할 때 시작된다."

### 선행 조건 (Preconditions): 시스템의 '시작 컨텍스트(Context)' 기술
유스케이스가 시작되기 위해 시스템이 갖춰야 하는 실제 상태나 제약 조건을 기술합니다. "A 유스케이스가 성공적으로 실행됨"과 같은 의존성 관점으로 적지 말고, **"로그인된 상태", "유효한 세션 보유"**와 같이 시스템이 처한 구체적인 데이터/환경 상태를 명시해야 합니다.

### 후행 조건 (Postconditions): 성공의 기준이 되는 '최종 상태' 요약
유스케이스가 종료되었을 때(성공뿐만 아니라 예외 흐름으로 인한 종료 포함) 시스템이 도달한 **최종 데이터 정합성 상태**를 요약합니다. 이는 다음 동작을 위한 트리거가 아니라, 해당 유스케이스가 완수되었음을 증명하는 '성공의 기준'이어야 합니다.

### 💡 [심화: 가치 중심 목표(Goal)와 중단점(Interruptible Point)의 관계]
에이전트 구현 시 가장 혼동하기 쉬운 부분은 '기술적 추적 ID'와 '실질적 유스케이스'의 차이입니다.

1. **가치 중심 목표 (Goal-Oriented Use Case)**:
   - 사용자가 시스템을 통해 얻는 **최종적이고 완결된 서비스 가치**입니다. (예: "지식 소비", "지식 자산화")
   - CRUD 하나하나가 아닌, 여러 상호작용이 모여 하나의 비즈니스 목표를 달성하는 '큰 덩어리(Chunk)'입니다.

2. **중단점 (Interruptible Point / Traceability ID)**:
   - 큰 흐름 속에서 **데이터가 저장되거나 시스템이 응답하는 구체적인 기술 지점**입니다.
   - `UC-U-01` (로그인), `UC-D-01` (목록 조회) 등과 같은 ID는 사실상 유스케이스가 아니라, **코드/테스트와의 추적성(Traceability)을 위해 유지하는 '중단점'**의 식별자입니다.
   - **표기 규격**: 중단점은 시나리오 내에서 관련 액션들을 묶는 **상단 제목(####)**으로 사용합니다. `{기능명 [추적ID]}` 형식의 제목 아래에 해당 중단점까지 도달하기 위한 액터와 시스템의 세부 상호작용 단계들을 기술하십시오.
   - **주의**: 구현 편의를 위해 이 ID들을 각각의 독립된 유스케이스로 간주하여 문서를 파편화하지 마십시오. 반드시 큰 '가치 중심 목표'의 시나리오 내부에 계층적으로 포함시켜야 합니다.
