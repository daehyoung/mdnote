# 01 요구사항 정의서 (Requirements Specification)

## 1. 개요
본 문서는 MarkDown Note System의 백엔드 시스템 및 유즈케이스 분석을 통해 도출된 시스템 요구사항을 정의한다.

## 2. 사용자 역할 및 권한 (Roles & Permissions)
* **Guest (비로그인 사용자)**: 퍼블릭으로 설정된 문서 목록 조회 및 문서 상세 내용 조회 가능.
* **User (일반 사용자)**: 
  - 회원가입, 로그인, 내 프로필 관리.
  - 문서 생성, 수정, 삭제 (본인 작성 문서에 한함).
  - 문서에 접근 권한 부여 (퍼블릭 읽기/쓰기, 그룹 읽기/쓰기).
  - 허용된 문서에 댓글 작성 및 파일 첨부.
* **Admin (관리자)**:
  - 시스템 내 모든 사용자 조회, 권한 변경, 부서 설정, 비밀번호 초기화, 계정 삭제.
  - 시스템 내 모든 문서(비공개 포함) 접근, 수정 및 삭제 권한(관리자 권한으로 열람) 보장.

## 3. 기능 요구사항 (Functional Requirements)

### 3.1 사용자 및 인증 (Auth & User Management)
* **REQ-U-01 (로그인)**: 사용자는 Username과 Password를 이용해 로그인할 수 있어야 하며, 성공 시 JWT 토큰(또는 Simple Token)을 발급받아야 한다.
* **REQ-U-02 (내 정보 관리)**: 사용자는 자신의 비밀번호 및 기본 테마(주/야간) 등을 조회하고 변경할 수 있어야 한다.
* **REQ-U-03 (사용자 관리-Admin)**: 관리자는 사용자를 신규 생성, 삭제, 상태 변경, 비밀번호 초기화 및 부서 배정을 수행할 수 있어야 한다.
* **REQ-U-04 (조직/부서 관리)**: 시스템은 사용자가 속한 부서(Department)를 계층적 트리 구조로 관리할 수 있어야 한다.

### 3.2 문서 관리 (Document Management)
* **REQ-D-01 (문서 생성/수정/삭제)**: 사용자는 마크다운 포맷의 문서를 생성, 수정, 삭제할 수 있어야 한다.
* **REQ-D-02 (문서 목록 및 검색)**: 문서는 카테고리, 상태(Draft, Published 등), 태그, 검색어(Title/Content)를 기준으로 필터링 및 페이징 조회될 수 있어야 한다.
* **REQ-D-03 (문서 권한 제어)**: 문서는 작성자에 의해 퍼블릭 읽기, 퍼블릭 쓰기, 그룹 읽기, 그룹 쓰기 여부가 설정되어야 하며, 인가된 사용자만 접근 가능해야 한다.
* **REQ-D-04 (첨부 파일)**: 사용자는 문서에 파일을 첨부할 수 있고, 첨부된 파일의 목록을 조회하고 다운로드/삭제할 수 있어야 한다.
* **REQ-D-05 (문서 메타데이터)**: 문서 생성일, 수정일, 작성자 정보가 시스템에 의해 자동 기록되어야 한다.

### 3.3 분류 및 태그 (Category & Tag Management)
* **REQ-C-01 (카테고리 분류)**: 문서는 트리 구조(Parent-Child)를 갖는 카테고리에 속할 수 있어야 한다.
* **REQ-C-02 (카테고리 관리)**: 카테고리의 생성, 수정, 삭제 및 이동이 가능해야 한다.
* **REQ-C-03 (다중 태그 연동)**: 문서는 다수의 태그(Tag)를 가질 수 있으며, 태그를 통한 문서 검색이 지원되어야 한다.

### 3.4 커뮤니케이션 (Communication)
* **REQ-CM-01 (댓글 작성)**: 사용자는 특정 문서에 댓글을 작성, 수정, 삭제할 수 있어야 한다.
* **REQ-CM-02 (댓글 목록 조회)**: 문서에 달린 댓글은 작성 시간순으로 조회되어야 한다.

## 4. 비기능 요구사항 (Non-Functional Requirements)
* **보안 (Security)**: 
  - API 통신은 JWT (또는 DevAuthToken) 기반 인증 필터를 거쳐야 한다.
  - 비밀번호는 BCryptPasswordEncoder를 통해 암호화되어 저장되어야 한다.
* **확장성 (Scalability)**:
  - 계층형 데이터(부서, 카테고리) 관리가 용이하도록 재귀적 관계 처리(Self-Referencing)가 고려되어야 한다.
* **유연성 (Flexibility)**:
  - 개발 환경과 운영 환경을 구분하기 위해 단순 인증 모드(`authType=simple`)와 JWT 모드(`authType=jwt`)가 환경변수에 의해 스위칭 가능해야 한다.
