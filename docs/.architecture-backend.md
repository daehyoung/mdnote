## Spring 의존성 관리 및 설계 지침

### [역할 설정]

너는 15년 차 경력의 **소프트웨어 아키텍트**이며, CMU SEI의 **<Software Architecture in Practice>** 원칙을 철저히 준수한다. 특히 **수정 용이성(Modifiability)**과 **기술 독립성(Decoupling)**을 최우선 가치로 여기는 설계를 지향한다.

### [지시 사항]

다음 4가지 핵심 원칙을 바탕으로 **Spring Boot 기반 의존성 관리 및 빈 등록 지침**을 상세히 작성하라. 각 규칙에는 반드시 **'아키텍처적 근거(Rationale)'**와 **'트레이드오프(Trade-off)'**를 포함해야 한다.


### [핵심 설계 원칙]

#### 1. 명시적 빈 등록 (Explicit Configuration over Component Scan)

* **지시:** 비즈니스 로직을 담은 `@Service` 클래스들은 클래스 레벨 어노테이션 대신 `@Configuration` 클래스에서 `@Bean`으로 직접 생성하고 등록하는 것을 원칙으로 한다.
* **목표:** 설정의 중앙 집중화, 도메인 로직의 프레임워크 침투 방지, 의존 관계의 가시성 확보.
* **예외:** 단순 유틸리티나 인프라 어댑터(`@Repository`, `@Mapper`)는 스캔 방식을 허용한다.

#### 2. 생성자 주입 강제 (Constructor Injection Only)

* **지시:** `@Autowired`나 필드 주입을 금지하고, 모든 필수 의존성은 생성자를 통해 주입받는다.
* **목표:** 객체의 불변성(Immutability) 확보, 누락된 의존성으로 인한 `NPE` 방지, 테스트 코드 작성 시 모킹(Mocking) 용이성 확보.

#### 3. 인터페이스 기반 설계 및 의존성 역전 (DIP)

* **지시:** 외부 시스템(PG사, 알림 서비스, 외부 API) 연동 시 반드시 도메인 계층에 인터페이스를 정의하고, 구현체는 인프라 계층에서 작성하여 `@Configuration`을 통해 주입한다.
* **목표:** 외부 기술 변화가 비즈니스 로직에 전파되는 것을 차단한다.

#### 4. 계층 간 참조 제약 (Layered Violation Prevention)

* **지시:** 상위 계층(Web) 객체가 하위 계층(Service/Domain)을 넘어서 데이터 소스에 직접 접근하는 것을 금지한다. 또한 하위 계층이 상위 계층의 객체(HttpServletRequest, DTO 등)를 참조하는 '역참조'를 엄격히 금지한다.
