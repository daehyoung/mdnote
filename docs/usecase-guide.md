 

# 📗 유스케이스(Use Case) 설계 가이드

## 1. 유스케이스란 무엇인가? (개념 정의)

유스케이스는 **"사용자(Actor)가 특정 목적을 달성하기 위해 시스템과 주고받는 대화의 시나리오"**입니다.

* **핵심 질문:** "사용자가 이 시스템을 통해 얻으려는 **최종 가치**가 무엇인가?"
* **오해 주의:** 유스케이스는 '로그인 버튼', '저장 기능' 같은 기능 목록이 아닙니다. 로그인은 목적을 위한 **수단**일 뿐이며, 진짜 유스케이스는 '상품 주문', '송금 완료'처럼 사용자에게 이득이 돌아가는 단위여야 합니다.

 

## 2. 유스케이스의 핵심 요소

유스케이스를 제대로 작성하려면 다음 세 가지를 반드시 기억해야 합니다.

1. **사용자 가치 (Value):** 이 과정을 마치면 사용자 손에 무엇이 남는가?
2. **중단점 (Interruptible Point):** 시스템이 일을 잠시 멈추고 사용자의 다음 행동을 기다리거나, 데이터를 안전하게 저장(Commit)하는 지점은 어디인가?
3. **시스템 경계 (Boundary):** 어디까지가 우리 시스템이 책임질 일이고, 어디서부터가 외부(사람/타 시스템)의 일인가?

4. **시스템 컨텍스트(Context)와 책임의 식별:** 시스템이 성공적으로 동작하기 위해 필요한 선행 조건과, 유스케이스가 성공적으로 마친 후의 후행 조건을 식별한다.

## 3. 유스케이스 기술서 작성법 (표준 구조)

하나의 유스케이스는 보통 다음과 같은 항목으로 구성됩니다.

| 항목 | 설명 | 예시 (상품 주문) |
| --- | --- | --- |
| **유스케이스명** | 사용자의 최종 목적을 능동형으로 표현 | 상품 주문하기 |
| **액터 (Actor)** | 시스템을 사용하는 주체 | 구매 회원 |
| **최종 가치** | 성공 시 사용자가 얻는 결과 | 원하는 물건의 결제를 완료함 |
| **선행 조건** | 시작 전 갖춰져야 할 상태(컨텍스트) | 로그인 완료, 장바구니에 상품 있음 |
| **기본 흐름** | 오류 없는 가장 이상적인 성공 경로 | 1. 결제 요청 -> 2. 재고 확인 -> 3. 결제 완료 |
| **예외 흐름** | 문제 발생 시 처리 경로 | 2-a. 재고 부족 시 '품절' 메시지 출력 |
| **사후 조건** | 종료 후 변화된 시스템 상태(성공의 기준) | 주문 내역 생성, 재고 수량 차감 |
| **필수 정보** | 트랜잭션 성공을 위해 필요한 데이터 (Payload) | 상품 ID, 결제 금액, 배송지 주소 |


## 4. 설계의 심화: 중단점과 콜래보레이션 (Collaboration)

단순히 글을 쓰는 것을 넘어, 실제 개발과 연결되는 지점입니다.

### 📍 중단점(Interruptible Point) 기반 설계

유스케이스의 각 단계는 **중단점**을 기준으로 나뉩니다.

* 사용자가 정보를 입력하고 '확인'을 누르는 순간 시스템은 내부적으로 복잡한 일을 처리하고 결과를 보여준 뒤 멈춥니다.
* 이 지점이 바로 **API가 설계되는 지점**이자, **DB 트랜잭션이 완료되는 지점**입니다.
* **필수 정보 (Collaboration 데이터/Payload):** UML에서 협력(Collaboration)을 완성하기 위해 필요한 데이터 세트를 의미합니다. 이 트랜잭션을 원자적으로 성공시키기 위해 액터가 시스템에 반드시 전달해야 하는 핵심 정보입니다.
* **상태 전이(State Transition)와 권한 변화:** 트랜잭션이 완료되면 시스템의 상태가 변경됩니다. 중요한 것은 이 시스템 상태의 변화가 이후 과정에서 액터의 역할(Role)이나 권한(Permission)을 동적으로 변화시키는 기준이 된다는 점입니다.

### 🤝 콜래보레이션(Collaboration)

중단점과 중단점 사이에서 시스템 내부의 여러 부품(객체)들이 서로 협력하는 것을 말합니다.

* "주문 완료"라는 중단점으로 가기 위해 `결제 모듈`, `재고 모듈`, `배송 모듈`이 서로 대화하는 과정이 콜래보레이션입니다.


## 5. 유스케이스가 개발 단계에 미치는 영향

유스케이스 기술서가 잘 작성되면 프로젝트 전체가 선순환됩니다.

1. **설계:** 중단점과 **필수 정보**를 보고 **API 엔드포인트(URL/Payload)**와 **데이터베이스 구조**를 잡습니다.
2. **개발:** 콜래보레이션 구조를 보고 어떤 **클래스와 함수**를 만들지 결정합니다.
3. **테스트:** 기술서의 '기본 흐름'과 '예외 흐름'이 그대로 **테스트 시나리오**가 됩니다.
4. **배포:** 유스케이스 단위로 기능을 묶어 사용자에게 점진적으로 배포(나선형 모델)할 수 있습니다.

 

## 6. 작성 시 주의할 점 (Checklist)
* [ ] **기능 분해를 하지 않았는가?** (로그인을 독립된 유스케이스로 뽑지 않았는지 확인)
* [ ] **사용자 관점인가?** (개발 용어가 아닌 사용자가 이해하는 용어 사용)
* [ ] **중단점이 명확한가?** (어디서 데이터가 저장되고 화면이 멈추는지 명확히 기술)
* [ ] **필수 데이터 항목(Collaboration Data)을 명시했는가?** (트랜잭션에 필요한 핵심 데이터 항목이 Payload로 기술되었는가?)
* [ ] **예외 상황을 고려했는가?** (사용자가 취소하거나 시스템 오류가 날 때의 대안)
* [ ] **선행 조건과 후행 조건을 명확히 기술했는가?** (성공을 위해 필요한 컨텍스트와 완료 후의 상태 보장)
* [ ] **시스템 경계(Boundary)를 명확히 기술했는가?** (누가 책임지는지, 장애 조치 주체는 누구인지 식별)
