# 🎨 프론트엔드 아키텍처 설계 지침 (Frontend Design Guidelines)

## 1. 프론트엔드 계층형 설계 (Layered Architecture)

GUI 시스템의 복잡도를 제어하기 위해 관심사를 4단계로 분리합니다.

| Layer | 역할 | 핵심 기술/규칙 |
| --- | --- | --- |
| **UI Component** | 데이터 시가화 및 사용자 이벤트 수신 | **Vuetify 3**, Presentational 컴포넌트 중심 |
| **Business Logic** | 도메인 로직 및 **공유 상태 관리** | **Pinia (Store)**, 프레임워크 독립적 순수 로직 |
| **Service (API)** | 백엔드 통신 및 데이터 매핑 | **Axios**, 전용 Service/Store를 통한 호출 |
| **Infrastructure** | 시스템 기반 설정 및 라이브러리 연동 | 라우터, **Vuetify Plugin**, 전역 인터셉터 |

 
## 2. API 및 테스트 전략 (Testability)

다양한 테스트 환경(Unit, Driver, Integration)에 대응할 수 있도록 의존성 주입(DI) 구조를 채택합니다.

### 가. 서비스 레이어 설계 원칙

* **인터페이스 정의:** 모든 API 서비스는 인터페이스를 먼저 정의하여 구현체와 분리합니다.
* **의존성 주입:** 컴포넌트는 구체 클래스가 아닌 인터페이스에 의존하며, 테스트 시 목적에 맞는 객체를 주입합니다.

### 나. 테스트 단계별 객체 활용

* **유닛 테스트 (Unit Test):** 로직이 포함된 함수나 컴포넌트를 고립시켜 검증합니다.
* **드라이버 테스트 (Driver Test):** **`createTestingPinia`**와 **`vi.mock`**을 사용하여 오프라인 환경에서 시나리오를 검증합니다.
* **통합 테스트 (Integration Test):** 실제 백엔드 API와 연동하여 전체 시스템 정합성을 확인합니다.

## 3. 전역 상태 및 에러 핸들링 (Cross-Cutting Concerns)

### 가. 공유 상태 관리 전략 (Pinia)
* **도메인별 스토어 분리:** `auth`, `document`, `organization` 등 도메인 단위로 스토어를 분리하여 관리합니다.
* **UI 전용 스토어:** 알림(Snackbar), 로딩 상태 등 전역 UI 제어를 위한 전용 스토어를 활용합니다.

### 나. 전역 인터셉터 (Global Interceptor)
* **보안:** 모든 요청에 JWT 토큰을 자동으로 주입합니다.
* **공통 에러 처리:** 401(만료) 시 자동 로그아웃 및 로그인 페이지 리다이렉트, 기타 에러 발생 시 `snackbarStore`를 통한 사용자 알림을 일괄 처리합니다.

## 3. UI 구조 및 내비게이션 (UX/UI Structure)

사용자 경험의 일관성과 데이터 접근성을 보장하는 설계입니다.

#### **마스터-디테일 패턴 (Master-Detail):**
* 좌측/상단에 목록(Master)을 두고, 선택 시 상세 내용(Detail)을 표시하는 구조를 표준으로 합니다.
* 목록 조회 API와 상세 조회 API를 명확히 분리하여 리소스 낭비를 방지합니다.


#### **라우터 기반 상태 관리 (Router-Driven State):**
* **북마크 가능성 (Bookmarkable):** 필터 조건, 페이지 번호, 선택된 카테고리 ID 등은 **URL Query String**(`?page=1&cat=5`)에 포함되어야 합니다.
* **데이터 복구:** 페이지 진입 시 URL 파라미터를 읽어 스토어 상태를 초기화함으로써 '뒤로 가기'나 'URL 공유' 시에도 동일한 컨텍스트를 유지합니다.

#### **서버 사이드 페이징 (Server-Side Paging)**
* 대량 데이터 조회 시 클라이언트 부하 방지를 위해 `page`, `size`, `sort` 파라미터를 서버에 전달하고 응답 값(`content`, `totalPages`)을 UI에 연동합니다.


## 4. 모듈화 및 컴포넌트 독립성 (Modularity)

시스템의 확장성을 위해 컴포넌트는 '레고 블록'처럼 독립적이어야 합니다.

* **독립적 배포 가능성:** 각 모듈은 자신의 서비스와 로직을 포함하며, 타 모듈에 대한 직접 참조를 금지합니다.
* **Props 기반 통신:** 부모-자식 간에는 Props와 Event(Emit)를 통해서만 데이터를 주고받습니다.
* **공통 컴포넌트 추출:** 3회 이상 재사용되는 UI 요소(버튼, 입력창 등)는 공통 라이브러리(UI Kit)로 분리합니다.
