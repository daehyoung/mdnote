# 🏛 시스템 아키텍처(Architecture) 설계 가이드

## 1. 아키텍처란 무엇인가? (개념 정의)

아키텍처는 **"시스템의 뼈대를 결정하는 가장 중요한 기술적 결정사항들의 집합"**입니다. 

* **핵심 질문:** "변화에 유연하게 대응하면서도, 시스템의 일관성과 성능을 어떻게 보장할 것인가?"
* **오해 주의:** 아키텍처는 단순히 '서버 구성도'가 아닙니다. 이는 데이터의 흐름, 모듈 간의 협력 방식, 그리고 기술적 제약 사항을 정의하는 **전략적 설계도**입니다.

---

## 2. 아키텍처 설계의 3대 원칙

1. **관심사의 분리 (Separation of Concerns):** 프레젠테이션(UI), 비즈니스 로직(Service), 데이터 접근(DB) 계층을 엄격히 분리하여 한 곳의 변화가 전체에 파급되지 않도록 합니다.
2. **기술적 정합성 (Technical Consistency):** 모든 모듈은 동일한 인증, 예외 처리, 데이터 변환 규칙을 따라야 합니다. (예: 모든 API는 JWT를 사용하며, 동일한 에러 객체 구조를 가짐)
3. **확장 가능한 단순성 (Scalable Simplicity):** 미래의 확장을 고려하되, 현재의 복잡도를 최소화합니다. (과도한 엔지니어링 지양)

---

## 3. 계층형 아키텍처 (3-Tier Architecture) 표준

본 프로젝트는 다음과 같은 계층 구조를 따릅니다.

| Layer | 역할 | 주요 기술/패턴 |
| --- | --- | --- |
| **Presentation** | 사용자와의 상호작용 및 UI 렌더링 | Vue 3, Vuetify, Pinia |
| **Application (API)** | 클라이언트 요청 처리 및 권한 검증 | Spring Boot, Controller, Security |
| **Domain (Business)** | 핵심 비즈니스 로직 및 객체 간 협력 | Service, Entity, JPA |
| **Data Source** | 데이터의 영구 저장 및 관리 | PostgreSQL, Repository |

---

## 4. 아키텍처 결정 사항 (ADR: Architecture Decision Records)

아키텍처 기술서(`03_architecture.md`) 작성 시 반드시 포함되어야 할 핵심 결정 사항들입니다.

1. **인증 전략:** 왜 JWT를 선택했는가? 만료 시간과 갱신(Refresh) 정책은 무엇인가?
2. **통신 프로토콜:** REST API 기반이며, 응답 포맷(JSON)의 표준은 무엇인가?
3. **보안 모델:** XSS, CSRF 방지 전략과 방화벽 정책은 어떻게 수립되었는가?
4. **데이터 무결성:** 유스케이스의 트랜잭션 바운더리를 DB단에서 어떻게 보장(ACID)하는가?

---

## 5. 아키텍처가 개발 단계에 미치는 영향

1. **구현:** 설계된 계층 구조를 위반하지 않도록 코드를 배치합니다. (예: Controller에서 직접 DB 조회 금지)
2. **품질:** 공통 모듈(Exception Handler, Logger 등)을 통해 시스템 전반의 운영 가시성을 높입니다.
3. **유지보수:** 컴포넌트 간의 결합도(Coupling)를 낮추어, 특정 기술(예: DB)의 교체가 용이해집니다.

---

## 6. 작성 시 주의할 점 (Checklist)

* [ ] **계층 간 침범이 없는가?** (상위 계층이 하위 계층에만 의존하는지 확인)
* [ ] **기술적 제약 사항이 명확한가?** (사용 가능한 라이브러리, 포트 번호, 성능 목표 등)
* [ ] **데이터 흐름도(Sequence Diagram)가 유스케이스와 일치하는가?**
* [ ] **보안 및 예외 처리 정책이 전사적으로 통일되었는가?**
* [ ] **인프라 구성(Docker, Network)이 논리적 아키텍처와 정합성을 이루는가?**
